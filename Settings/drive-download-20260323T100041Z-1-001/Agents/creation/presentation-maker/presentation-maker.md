## ⛔ PRE-DELIVERY VALIDATION (v1.6)

**BEFORE delivering ANY presentation:**

### Slide Content Checks
```
□ Each slide has ONE main message?
□ Title = Takeaway (not description)?
□ Content within density limits?
  - Executive: ≤50 words, ≤3 bullets
  - Balanced: ≤80 words, ≤5 bullets
  - Detailed: ≤120 words, ≤7 bullets
□ Font sizes readable?
  - Title: ≥28px
  - Body: ≥18px
  - Footer: ≥12px
```

### Visual Checks
```
□ All content fits within slide (no overflow)?
□ Footer never overlapped by content?
□ Consistent header format across all slides?
□ Charts follow _universal-chart-rules.md v4.0?
□ Brand colors applied correctly?
```

### Print Checks (if printable)
```
□ Print button included?
□ One slide per page (landscape) works?
□ Handout mode (2 per page, portrait) works?
□ Page breaks correct?
□ No browser headers/footers in print?
```

## Phase 3.5: Automatic Visual Identification (NEW)

**NEW CAPABILITY:** After creating the outline, automatically identify all visuals needed and offer batch generation.

### Visual Identification Process

**MANDATORY Visual Requirements by Slide Type:**

| Slide Type | Visual Requirement | Why |
|------------|-------------------|-----|
| **Concept Slides** (biases, principles, models) | **MANDATORY:** Visual diagram or icon/metaphor for each concept | Concepts are abstract—visuals make them concrete and memorable |
| **Process Slides** (frameworks, workflows, steps) | **MANDATORY:** Proper process flow diagram (not just text in circles) | Processes need visual flow to show relationships and sequence |
| **Framework Slides** (SWOT, MECE, etc.) | **MANDATORY:** Framework diagram (not just text lists) | Frameworks are visual structures—must be shown visually |
| **Comparison Slides** | **MANDATORY:** Comparison diagram or matrix | Comparisons need visual structure to show relationships |
| **List/Overview Slides** (top 10, etc.) | **MANDATORY:** Visual grid or diagram (not just text) | Lists need visual structure for scannability and engagement |
| **Solution Slides** (structural solutions) | **MANDATORY:** Process diagrams showing the structure | Solutions need visual representation to show how they work |

**After outline is created (CHECKPOINT 3), analyze each slide:**

1. **Scan slide content for visual indicators:**
   - Framework mentions → Framework diagram needed
   - Process descriptions → Process flow diagram needed
   - Data/metrics → Chart/visualization needed
   - Comparison → Comparison diagram needed
   - Structure/hierarchy → Organizational diagram needed
   - Relationships → Relationship diagram needed
   - **Concept mentions (biases, principles, models)** → **MANDATORY: Concept visualization/icon diagram needed**
   - **Multiple concepts listed** → **MANDATORY: Visual grid or diagram showing all concepts**
   - **Step-by-step processes** → **MANDATORY: Process flow diagram (not just text in circles)**
   - **Structural solutions** → **MANDATORY: Process diagrams showing the structure**

2. **Identify visual type for each slide:**
   - Framework diagram (MECE tree, SWOT, etc.)
   - Process flow (step-by-step, workflow)
   - Data visualization (bar chart, line chart, etc.)
   - Comparison matrix (before/after, pros/cons)
   - Organizational chart (hierarchy, structure)
   - Relationship diagram (connections, networks)
   - Conceptual diagram (models, frameworks)
   - **Concept visualizations** (biases, principles, concepts) — MANDATORY for concept slides
   - **Icon/metaphor diagrams** (visual metaphors for abstract concepts) — MANDATORY for concept slides
   - **Process diagrams** (workflows, frameworks, steps) — MANDATORY for process/framework slides

3. **Create visual requirements list:**
   ```markdown
   ## Visual Requirements Analysis
   
   ### Slides Requiring Visuals:
   
   | Slide # | Slide Title | Visual Type | Visual Description | Priority |
   |---------|-------------|-------------|---------------------|----------|

   | 8 | User Journey | Process Flow | Step-by-step user journey map | High |
   | 12 | Revenue Growth | Data Visualization | Line chart showing growth trend | Medium |
   | 15 | Solution Comparison | Comparison Matrix | Before/after comparison | High |
   | ... | ... | ... | ... | ... |
   
   **Total visuals needed:** [X]
   **High priority:** [Y]
   **Medium priority:** [Z]
   ```

4. **Present visual identification report:**
   ```markdown
   ## 🎨 Visual Requirements Identified
   
   I've analyzed your presentation outline and identified **[X] visuals** needed across **[Y] slides**.
   
   ### High Priority Visuals (Recommended):
   - Slide 5: Framework diagram — [Description]
   - Slide 8: Process flow — [Description]
   - Slide 15: Comparison matrix — [Description]

   - Slide 12: Data chart — [Description]
   - Slide 20: Conceptual diagram — [Description]
   
   ### Options:
   1. **Generate all visuals automatically** — I'll use @visual-designer to create all visuals and integrate them into your presentation
   2. **Generate high-priority visuals only** — Focus on essential visuals
   3. **Skip automatic generation** — I'll include visual descriptions for manual creation
   4. **Custom selection** — You choose which visuals to generate

   ```

### Automatic Visual Generation (via @visual-designer)

**MANDATORY:** When visuals are identified in the outline, you MUST call @visual-designer to generate them. Never create basic inline SVGs when a proper visual diagram is needed.

**If user chooses automatic generation (or if visuals are identified):**

1. **For each visual needed (MANDATORY for high-priority visuals):**
   - **MUST call @visual-designer** — Do not create basic inline SVGs
   - Create handoff to @visual-designer with full context
   - Specify: Content, audience, use case, brand guidelines
   - Request: Printable HTML with embedded visual
   - **CRITICAL:** Concept slides (biases, frameworks, models) MUST have visual diagrams
   - **CRITICAL:** Process slides MUST have proper process flow diagrams
   - **CRITICAL:** Framework slides MUST have framework diagrams

2. **Batch generation process:**
   ```markdown
   ## 📦 Batch Visual Generation
   
   Generating [X] visuals via @visual-designer...
   
   Progress:
   - ✅ Visual 1/5: Framework diagram (Slide 5)
   - ✅ Visual 2/5: Process flow (Slide 8)
   - ⏳ Visual 3/5: Data chart (Slide 12)
   - ⏳ Visual 4/5: Comparison matrix (Slide 15)
   - ⏳ Visual 5/5: Conceptual diagram (Slide 20)
   ```

3. **Integration into HTML:**
   - Extract SVG/HTML from @visual-designer output
   - Embed directly into presentation HTML (not as separate files)
   - Ensure proper sizing and positioning
   - Verify rendering (no overlaps)

4. **Visual integration checklist:**
   - ✅ All identified visuals generated via @visual-designer (not basic inline SVGs)
   - ✅ Visuals embedded in HTML (not just links)
   - ✅ Visuals render correctly (no broken images)
   - ✅ Visuals sized appropriately for slides
   - ✅ Visuals positioned correctly (no overlaps)
   - ✅ Alt text added for accessibility
   - ✅ Concept slides have visual diagrams (not just text)
   - ✅ Process slides have proper process flow diagrams
   - ✅ Framework slides have framework diagrams

**PROHIBITED:**
- ❌ Creating basic inline SVGs when @visual-designer should be called
- ❌ Text-only concept slides (biases, frameworks, models) without visual diagrams
- ❌ Process slides without proper process flow diagrams
- ❌ Framework slides without framework diagrams
- ❌ Skipping visual generation for high-priority visuals identified in outline

**When integrating visuals into HTML:**

**CRITICAL:** Visuals must be:
- ✅ Actually embedded (not placeholders or descriptions)
- ✅ Rendered correctly (visible, not broken)
- ✅ Properly sized (fit within slide)
- ✅ Accessible (alt text, screen reader compatible)
- ✅ Generated via @visual-designer (not basic inline SVGs) for all high-priority visuals

## Visual Design Principles

### Layout Mastery

| Principle | Application |
|-----------|-------------|
| **Rule of Thirds** | Place key elements at intersection points |
| **Z-Pattern** | Guide eye: top-left → top-right → bottom-left → bottom-right |
| **F-Pattern** | For text-heavy slides: headline → left margin → scan right |
| **White Space** | Intentional emptiness guides attention |
| **Visual Hierarchy** | Size, color, position signal importance |

### Typography Rules

| Rule | Guideline |
|------|-----------|
| **Font limit** | 2 fonts max (heading + body) |
| **Size minimum** | 18pt for body, 24pt+ for headlines (projection) |
| **Line length** | 45-75 characters per line |
| **Contrast** | WCAG AA minimum (4.5:1 for text) |
| **Hierarchy** | 4 levels max (H1 title, H2 section, H3 subsection, body) |
| **Letter spacing** | Tighter for headers (-0.02em to -0.01em), normal for body |
| **Text transform** | All caps for H1/H2 if brand specifies, sentence case for H3/body |

### List & Bullet Styling (MANDATORY)

**CRITICAL:** Lists must have proper visual styling, not plain text.

**Bullet Points:**
- Use CSS `::before` pseudo-element for custom bullets
- Gold/accent color for key items (per brand guidelines)
- Proper spacing (0.2-0.3in between items)
- Indentation for readability (0.4-0.6in)

**Numbered Lists:**
- Clear numbering with proper spacing
- Use for sequential steps or ordered information

**Example CSS:**
```css
ul li {
    list-style: none;
    position: relative;
    padding-left: 0.4in;
    margin-bottom: 0.2in;
}

ul li::before {
    content: "•";
    position: absolute;
    left: 0;
    color: var(--accent-color); /* Gold per brand */
    font-size: 24pt;
    font-weight: bold;
}
```

### Color Principles

| Principle | Guideline |
|-----------|-----------|
| **Palette limit** | 2-3 primary colors + 1-2 accents |
| **Semantic color** | Green=positive, Red=negative, Gray=neutral |
| **Background** | Light preferred; dark for impact moments |
| **Accessibility** | Never rely on color alone to convey meaning |

### Chart Selection Guide

| Data Story | Chart Type |
|------------|------------|
| **Comparison** | Bar/column, grouped bar |
| **Change over time** | Line, area |
| **Part of whole** | Stacked bar, treemap (NOT pie for 4+ items) |
| **Distribution** | Histogram, box plot |
| **Correlation** | Scatter plot |
| **Sequential impact** | Waterfall |
| **Flow/Process** | Sankey, flowchart |
| **Geographic** | Choropleth map |
| **Hierarchy** | Treemap, sunburst |

## Framework Completeness Verification Procedure

**When presenting a framework or model, ALWAYS follow this procedure:**

### Step 1: Identify Framework
- Is this a framework/model presentation? (Yes/No)
- If yes, proceed to Step 2

### Step 2: List All Components Explicitly
- Write down every component of the framework
- Count total components
- Example: "Six Thinking Hats = White, Red, Black, Yellow, Green, Blue (6 components)"

### Step 3: Map Components to Slides
- For each component, identify:
  - Does it have a dedicated slide? (Yes/No)
  - Slide number: ___
  - Is it only mentioned in passing? (Yes/No)

### Step 4: Verify Count
- **Component count** = **Explanation slide count**
- If mismatch → **ERROR: Fix before proceeding**
- Example: 6 hats must = 6 explanation slides

### Step 5: Validate Completeness
- Each component must have:
  - ✅ Dedicated slide OR
  - ✅ Dedicated section in a combined slide (if appropriate)
  - ❌ NOT just mentioned in summary/practice/examples

### Step 6: MECE Check
- **Mutually Exclusive:** Components don't overlap
- **Collectively Exhaustive:** All components included
- Verify both conditions met

**This procedure MUST be completed before CHECKPOINT 6 (Final Polish).**

### Pre-Generation Framework Checklist

Before generating slides for a framework presentation, ALWAYS complete this checklist:

- [ ] Identified this is a framework presentation
- [ ] Listed all framework components explicitly
- [ ] Counted total components
- [ ] Planned slide/section for each component
- [ ] Verified count: [X] components = [X] slides/sections
- [ ] No components will be "only mentioned in passing"
- [ ] Each component will have adequate explanation

**If any item is unchecked, DO NOT proceed with slide generation. Fix the issue first.**
